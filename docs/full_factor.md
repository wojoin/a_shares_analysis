全量因子分析可以按“`板块层 + 个股层 + 组合层`”三层来做，这样既能解释市场状态，也能直接转成交易动作。

**1) 因子全集（建议）**
- `技术因子`：趋势（MA5/20/60）、动量（MACD柱/动量、RSI、KDJ）、波动（ATR%、布林带宽）、风险距离（止损空间%）。
- `资金因子`：成交额、换手率、成交额分位、板块成交占比、主力净流入、北向净流入、机构持仓变化。
- `基本面因子`：营收/利润增速、毛利率、ROE、现金流、资产负债率、应收周转、研发强度。
- `估值因子`：PE/PB/PS 当前值、历史分位（1Y/3Y）、PEG、同业相对估值偏离。
- `产业链因子`（CPO特有）：环节位置（芯片/器件/模块/设备）、客户结构（海外云厂商占比）、订单能见度、价格趋势。
- `事件因子`：财报超预期、指引上修、订单公告、政策催化、减持/解禁、监管或贸易限制。
- `情绪因子`：涨停家数、连板强度、题材热度、舆情热词强度。

**2) 评分框架（0-100）**
- 板块总分：`资金(30) + 扩散(20) + 动量(20) + 估值位置(10) + 产业景气(10) + 事件情绪(10)`
- 个股总分：`技术(25) + 资金(20) + 基本面(20) + 估值(10) + 产业链地位(15) + 事件(10) - 风险惩罚(0~25)`
- 输出分层：`S/A/B/C`，并给 `entry_flag`、`risk_flag`、`conviction`（置信度）
- 风格切换：`aggressive / balanced / defensive` 只改权重与阈值，不改底层因子定义。

**3) 日更流程（可执行）**
- 盘后更新：行情与资金因子日更；事件因子实时或日更；基本面/估值周更或财报后更新。
- 因子标准化：全部转成 `z-score` 或 `0-1分位`，避免量纲冲突。
- 缺失值处理：缺失不直接记 0，按“行业中位数回填 + 降低该因子权重”。
- 稳定性控制：单因子分值做 `EMA平滑`，防止一天噪声导致频繁跳档。

**4) 结果呈现（你当前脚本可扩展方向）**
- `板块评分卡`：总分、分项分、状态（进攻/观察/防守）、关键驱动因子。
- `成分股评分榜`：总分+分项雷达（技术/资金/基本面/估值/事件），并标注入场与风险信号。
- `风险提示榜`：高波动、高估值、高解禁、事件负面预警。

**5) 回测与验收（上线前必做）**
- 看分层有效性：`S组未来5/10/20日收益` 是否稳定高于 `B/C组`。
- 看换手与回撤：信号频率、胜率、盈亏比、最大回撤是否符合风格预期。
- 看稳健性：牛/震荡/熊三种区间是否都可解释，不只在单一行情有效。

## CPO 板块补充因子

以下因子针对 CPO 板块的特殊性补充，通用框架未覆盖或覆盖不足。

### 1. 云厂商 CapEx 节奏因子（重要程度：★★★★★）

- **为什么重要**：CPO 需求几乎完全由 AWS / Azure / GCP / Meta 资本开支驱动，是板块景气度的最强领先指标。云厂商 CapEx 指引上调往往领先 CPO 个股上涨 1–2 个季度。
- **数据来源**：云厂商季度财报后手动更新，写入 `config.json` → `cpo_cloud_capex`（示例：`{“level”: “high”, “yoy_growth”: 0.45, “updated”: “2026-04”}`）。
- **建议**：纳入板块评分”产业景气”子项，权重建议 5–8 分（从现有 10 分中分配）。`level=high` 时满分，`medium` 打 6 折，`low` 打 3 折。季度更新一次，缺失时用上季度值并降低该因子权重。

### 2. 产业链环节位置量化方法（重要程度：★★★★☆）

- **为什么重要**：激光芯片（上游）、CPO 模块（中游）、测试设备（下游）的毛利率和议价权差异大；同样的技术指标，上游芯片公司的”超买”含义与中游模块厂完全不同。
- **数据来源**：在 `manual_overrides` 中按股票代码配置 `chain_position`，取值 `upstream` / `mid` / `downstream`。可一次性配置所有已知 CPO 成分股，之后无需频繁更新。
- **建议**：作为个股”产业链地位”子项的基础权重乘数，`upstream` × 1.2，`mid` × 1.0，`downstream` × 0.9。缺失时默认 `mid`，不阻塞评分。

### 3. 贸易管制风险因子（重要程度：★★★★★）

- **为什么重要**：美国出口管制是 CPO 板块最大尾部风险，单一事件可导致板块单日 -10%+。该风险不适合用打分模型平滑处理，应直接触发硬性风险标记。
- **数据来源**：手动配置 `config.json` → `trade_risk_level`，取值 `low` / `medium` / `high`。事件发生后即时更新，平时维持 `low`。
- **建议**：`high` 时强制板块及所有个股 `risk_flag=True`，在风险提示榜独立列出，不受打分模型影响。`medium` 时风险惩罚上限提高 5 分（即 `risk_cap` +5）。

### 4. 技术路线 / 商业化阶段因子（重要程度：★★★☆☆）

- **为什么重要**：CPO 成分股中研发期与量产期的估值逻辑完全不同——研发期公司高 PE 可接受，量产期公司看 ROE 和交付节奏。混用同一套打分会系统性高估研发期标的。
- **数据来源**：`manual_overrides` 中配置 `commercialization_stage`，取值 `rd`（研发）/ `pilot`（试产）/ `mass`（量产）。
- **建议**：作为置信度（`conviction`）修正系数：`mass` × 1.0，`pilot` × 0.85，`rd` × 0.70。不影响原始评分，只影响最终置信度输出。缺失时默认 `pilot`。

### 5. 客户集中度风险因子（重要程度：★★★★☆）

- **为什么重要**：部分 CPO 个股前两大客户占收入 70%+，单客户依赖在贸易管制或客户砍单时会放大下行风险，应计入风险惩罚。
- **数据来源**：`manual_overrides` 中配置 `top2_customer_pct`，0–1 浮点数（如 `0.72` 表示前两大客户占 72% 收入）。年报披露后更新一次。
- **建议**：`> 0.70` 时追加 −5 风险惩罚分；`> 0.85` 时触发 `risk_flag`。缺失时不加惩罚，但在输出中标注”客户集中度数据缺失”提示用户补充。

### 关于自动抓取 vs 手动配置

| 因子 | 抓取方式 | 更新频率 |
|---|---|---|
| 主力净流入 | akshare 自动抓取，日缓存 | 每日自动 |
| 北向净流入 | akshare 自动抓取，日缓存 | 每日自动 |
| 云厂商 CapEx | 手动配置 `config.json` | 季度（财报季后） |
| 产业链环节位置 | 手动配置 `manual_overrides` | 一次性，成分股变化时更新 |
| 贸易管制风险 | 手动配置 `config.json` | 事件驱动，平时无需改动 |
| 商业化阶段 | 手动配置 `manual_overrides` | 半年度，量产节点变化时更新 |
| 客户集中度 | 手动配置 `manual_overrides` | 年度（年报后） |

**设计原则**：系统不因手动因子缺失而报错或阻塞，缺失时使用中性默认值并在输出中明确告知用户该因子的重要程度和补充建议。
